# Self-Reflection

When I started this assignment, I assumed the hardest part would be the UI. It turned out to be everything I couldn't see: the network layer, the JSON parsing, and understanding why my app crashed before anything even appeared on screen.

The biggest early challenge was getting Retrofit and Gson to cooperate. My first data classes didn't match the structure of the OpenWeatherMap response, so the app would either crash or silently show nothing. Debugging this taught me to actually read the raw JSON before writing a single Kotlin class, and to use @SerializedName when the API's naming (like feels_like) doesn't match Kotlin conventions. It sounds obvious in hindsight, but I lost a decent chunk of time to it, and it permanently changed how I approach any API — response first, code second.

Working with MVVM was the other real learning curve. At first, the ViewModel and Repository felt like pointless ceremony for such a small app — why not just call the API from the composable? But the value became obvious the moment I started enhancing the app. When I redesigned the weather screen into a card layout with a weather icon and pull-to-refresh, I didn't have to touch the networking code at all. The separation I'd grudgingly set up earlier meant the UI could change freely while the data layer stayed stable. That was the point in the project where the architecture stopped being a requirement and started being a tool.

On design, I made a deliberate decision to keep the app to two screens: the weather display and a settings screen for city and temperature units. I considered adding more features early on, but I'd rather submit a small app that works properly than an ambitious one that half works. The features I did add — pull-to-refresh, a feels-like temperature, humidity and wind cards — came from thinking about how I actually use weather apps on my own phone, rather than what would look impressive in a screenshot.

I used AI assistance (Claude) during development, mainly for refactoring the main screen into the card-based layout and as a second pair of eyes on the code. It flagged things I would have missed, like the API key being hardcoded in the repository — fine for a course project, but a bad habit for anything real. I was careful to treat it as a reviewer rather than an author: I read and understood every suggested change before accepting it, and the core decisions — the app concept, the architecture, the feature scope — were mine. The most useful part wasn't the code it produced but the questions it forced me to answer about my own choices.

If I were to continue this project, the first thing I'd fix is persistence. The city and unit settings currently reset every time the app restarts, which is the app's most obvious flaw as a daily-use utility — DataStore would solve it cleanly. I'd also add a 5-day forecast using the forecast endpoint, and proper error states that distinguish "no internet" from "city not found."

Overall, the assignment shifted how I think about mobile development. I came in thinking of an app as screens; I'm leaving thinking of it as layers, where the screens are just the thinnest one on top.
